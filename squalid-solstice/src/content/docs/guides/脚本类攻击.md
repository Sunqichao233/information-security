# 脚本类攻击

## 跨站脚本攻击（XSS）

当 Web 页面存在“输入并回显”的表单或处理逻辑时，攻击者可注入恶意脚本（简易程序），并借助其他页面或站点触发执行。

攻击后果通常包括：

- 窃取 Cookie 等敏感数据
- 篡改页面显示内容
- 将用户诱导到钓鱼网站

该攻击的常用缩写为 `XSS`。

### 术语拆解与区别

- 跨站（Cross-site）：指“横跨多个 Web 站点/页面”的上下文。
- 脚本（Scripting）：指脚本代码被解析并执行的处理过程。

组合起来，XSS 的核心不是“服务器被执行脚本”，而是恶意脚本在受害者的客户端（浏览器侧）被执行。

也就是说，XSS 的关键点是：攻击代码借助网页传播，但执行位置通常在用户浏览器端。

### 典型考点（XSS 出题思路）

当网页存在“把用户输入原样显示”的表单或处理流程时，攻击者可插入恶意脚本，进而窃取 Cookie 等数据。

其本质脆弱点在于：网站即使并非故意，也会把“包含脚本的数据”发送到用户浏览器并执行，导致客户端侧被利用。

## Cookie（相关概念）

Cookie 是 Web 服务器临时写入用户浏览器的一段信息。

用户下次访问该服务器时，服务器可利用 Cookie 识别用户状态（例如会话识别等）。

## 驱动下载攻击（Drive-by Download）

驱动下载攻击是指：攻击者先在网站中植入恶意程序，用户仅仅通过浏览器访问该站点，就可能在不知情情况下被利用浏览器或插件漏洞，导致恶意软件被下载或执行。

典型特征是“只要访问网页就可能中招”，不一定需要用户主动点击下载按钮。

## 跨站请求伪造（CSRF）

跨站请求伪造是指：用户访问恶意网站后，在不知情情况下由浏览器向目标站点发出恶意请求（如脚本触发请求），从而借用用户已登录身份执行非本人意图的操作。

常见影响包括：

- 在论坛/留言板被冒名发帖
- 在已登录的网购网站中被恶意下单

该攻击常用缩写为 `CSRF`。

## XSS 与 CSRF 的区别

两者都可能涉及恶意脚本或恶意请求，但关键区别在“执行/发送的位置”：

- XSS：恶意脚本在用户浏览器内执行（利用页面注入）。
- CSRF：利用用户已登录状态，向目标网站发送伪造请求（利用身份上下文）。

## SQL 注入（SQL Injection）

攻击者在 Web 输入项中注入数据库操作语句（SQL 语句），使后端数据库执行非预期操作，从而导致数据被窃取、篡改或越权访问。

## OS 命令注入（OS Command Injection）

攻击者通过表单等输入点注入可触发系统命令执行的恶意字符串，使存在漏洞的站点在服务器侧执行非授权 OS 命令，造成非法操作。

## 目录遍历（Directory Traversal）

攻击者通过文件名/路径参数构造特殊路径，访问管理员原本不希望公开的文件或目录，从而读取受限资源。

## 缓冲区溢出（Buffer Overflow）

攻击者向输入项提交超过缓冲区容量的数据（缓冲区是用于暂存输入数据的内存区域），导致程序出现非预期行为，进而可能被利用。

注意：严格来说，缓冲区溢出不属于“脚本攻击”本身，而是更底层的内存安全漏洞利用。

## 各攻击的对策（1）

### 输入表单对策（面向 XSS 与 SQL 注入）

#### 转义处理（XSS 对策）

对脚本语言中具有特殊含义的字符（如 `&`、`"`、`<`、`>` 等）进行安全替换，避免用户输入被当作脚本执行。

这类“将危险字符无害化”的处理也常被称为清理（Sanitizing）的一部分。

#### 占位符（Placeholder，SQL 注入对策）

将用户输入作为“后续绑定的参数”处理，并与 SQL 语句结构分离（参数化查询/预编译语句）。

这样即使输入了恶意字符串，也会被当作数据值，而不是 SQL 命令本体执行。

## 各攻击的对策（2）

### OS 命令注入对策

从设计上避免 Web 应用直接调用 OS 命令；如必须调用，应采用严格白名单与最小权限控制，并与用户输入彻底隔离。

### 目录遍历对策

不要直接接收客户端传入的路径/文件名；建议只接收 `file_id` 这类标识，再由服务器端映射到允许访问的固定路径。

### 缓冲区溢出对策

对输入数据实施长度校验与边界检查；优先使用不允许直接危险内存操作或具备内存安全机制的开发语言与运行时。
